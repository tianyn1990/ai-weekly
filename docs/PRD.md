# AI 周报项目 PRD（v0.4）

## 0. 文档状态
- 版本：v0.3
- 状态：M4 已实现基线
- 更新时间：2026-03-06

## 1. 目标与读者
- 主要读者：前端开发团队、Agent 开发团队。
- 项目目标：
  1. 建立 AI 领域持续追踪机制，覆盖技术趋势、优秀产品/开源项目、行业动态。
  2. 输出可读、可验证、可追溯的日报/周报。
  3. 通过真实项目实践系统学习 LangChain/LangGraph。
- 写作风格：混合风格，侧重技术深度，保持工程可落地视角。

## 2. MVP 范围（冻结）
### 2.1 本期必须完成
- 每日凌晨自动采集（北京时间）。
- 每天 09:00 产出日报草稿（北京时间）。
- 每周一 09:00 产出周报待审核版本（北京时间）。
- 周报人工审核窗口：09:00 - 12:30（北京时间）。
- 若周一 12:30 前未完成审核，系统自动发布当前周报版本。
- 发布渠道：Markdown 仓库。
- 历史报告可检索（按日期/周期/分类）。

### 2.2 本期暂不做（非目标）
- 多租户。
- 复杂前后端管理系统。
- 移动端 App。
- 实时流式处理。

## 3. 内容策略
- 内容取向：工程实践优先，同时覆盖行业重要动态。
- 覆盖优先级：信息全覆盖优先，但必须分级展示重要性/推荐程度。
- 语言策略：
  - 信息源：中英混合。
  - 报告正文：简体中文（专业术语保留英文）。

## 4. 篇幅与结构要求
### 4.1 日报
- 可读时长：4-12 分钟。
- 顶部必须有「重点推荐 / 今日精华」区块。
- 主体需按分类分组，给出简要分析与行动建议。

### 4.2 周报
- 可读时长：12-20 分钟。
- 顶部必须有「本周重点推荐 / 精华摘要」区块。
- 主体需按分类分组，强调趋势、影响范围、工程启发。

## 5. 数据与来源策略
- 首批来源：固定 5-10 个白名单来源（可迭代扩展）。
- 黑名单：当前为空，但系统必须支持。
- 每条结论必须可追溯到来源证据（链接或引用片段）。

## 6. 质量验收标准（MVP）
- 每期有效条目数：>= 20（周报）。
- 重复率：< 10%。
- 无来源断言容忍度：0。
- 核心排序指标：重要性、影响范围、创新性。
- 强约束问题：
  - 与前端/后端/Agent 开发者相关性过低。
  - 仅搬运信息，缺少分析处理与评估。

## 7. 自动化与审核策略
- 自动化目标：采集、去重、分类、排序、草稿生成全自动。
- 人工介入：周报至少支持「大纲审核 + 最终稿审核」。
- 超时兜底：超过审核截止时间（周一 12:30）自动发布。
- 审核后重发：支持人工复审后再次发布修订版。
- 审核通知渠道：默认飞书（Feishu），支持在关键节点推送待办与截止提醒。
- 审核触发方式：支持通过飞书完成审核动作输入（通过/打回/需修改）。
- 审核意见回流：回流不等于取消，系统需支持在原始内容基础上进行结构化调整：
  - 新增/删除候选条目
  - 新增主题或搜索词
  - 数据源开关与权重调整
  - 分类/排序权重调整
- 打回策略：支持将周报从终稿审核打回到修订阶段，修订后再次进入审核。

### 7.1 审核动作字典（M3.2 必须）
- `approve_outline`：大纲通过，进入终稿审核。
- `approve_final`：终稿通过，立即发布。
- `request_revision`：要求修订，进入修订分支后再次终稿审核。
- `reject`：终止本轮发布尝试，保留产物与审计记录。

### 7.2 审核意见回流指令（M3.3 必须）
- `candidate_additions`：人工新增候选条目（title/link/snippet/categoryHint）。
- `candidate_removals`：人工删除候选条目（id/link）。
- `new_topics`：新增主题词，用于后续检索和分组强化。
- `new_search_terms`：新增搜索词，用于采集阶段扩展覆盖。
- `source_weight_adjustments`：来源权重调整（sourceId -> weight）。
- `ranking_weight_adjustments`：分类/重要性权重调整。
- `editor_notes`：仅展示，不自动执行。

### 7.3 审核回流边界（防止歧义）
- 回流默认语义是“修订增强”，不是“取消流程”。
- `request_revision` 必须保留原始证据链并产出修订日志。
- `reject` 不删除历史文件，必须可追溯是谁在何时拒绝。
- 所有自动修订动作必须可回放（输入指令 + 输出差异）。

## 8. 成本与性能策略
- 当前阶段不设置硬预算上限，但系统需保留预算开关与监控点。
- 当前阶段不设置硬耗时上限，但系统需记录各阶段耗时。
- 失败重试策略：激进（多次重试 + 失败告警）。
- watchdog 分布式互斥：当前阶段暂缓，仅保留单机 lock 方案。

## 9. 技术约束
- 模型供应商偏好：OpenAI、Anthropic、MiniMax。
- 必须支持后续模型切换。
- 部署环境：本地定时任务。

## 10. 学习交付要求
每次迭代必须同步提供：
1. 流程图（Graph 视角）。
2. 源码导读（模块职责 + 关键接口）。
3. 复盘文档（决策、取舍、后续优化点）。

## 11. 里程碑建议
- M1：可跑通的日报/周报骨架（采集->处理->草稿输出）。
- M2：人工审核闭环 + 自动发布策略 + 历史检索页。
- M3.1：watchdog 可靠性（锁、重试、summary）【已完成】。
- M3.2：飞书协同审核闭环（通知、动作回写、截止提醒）。
- M3.3：审核意见回流修订（新增/删除候选、主题词/搜索词、权重调整）。
- M4：审核指令与历史数据持久化升级（DB/API + 审计 + 并发控制）【已完成】。
- M5：LLM 增强（总结优先，逐步覆盖分类/打标/排序辅助）。
- M6：月报/季报/年报聚合与趋势分析。

## 12. 可测试验收清单（新增）
### 12.1 M3.2（飞书协同）验收
- [ ] 周报待审核生成后，飞书收到待办通知（包含 reportDate、审核截止、review 文件路径）。
- [ ] 审核人在飞书执行 `approve_outline` 后，状态从 `outline_review` 进入 `final_review`。
- [ ] 审核人在飞书执行 `approve_final` 后，状态变为 `approved` 且产物发布。
- [ ] 截止前至少有一次飞书提醒；截止后发布结果可回执到飞书。
- [ ] 飞书链路不可用时，CLI fallback 仍可完成审核与发布兜底。

### 12.2 M3.3（回流修订）验收
- [ ] `request_revision` 可附带结构化指令并触发修订流程。
- [ ] 新增/删除候选条目后，修订稿内容与指标同步变化。
- [ ] 主题词/搜索词/权重调整会影响下一次修订结果排序。
- [ ] 修订完成后自动回到终稿审核阶段。
- [ ] `reject` 仅终止本轮发布尝试，不删除历史待审核产物。

### 12.3 M4（存储升级）验收
- [ ] 审核指令写入 DB/API 后与文件模式行为一致。
- [ ] 审核动作带审计字段（operator/source/timestamp/traceId）。
- [ ] 并发写入场景不会产生脏状态（至少有版本冲突保护）。

### 12.4 M5（LLM 增强）验收
- [ ] 新增总结节点后，报告保留来源证据且无无源断言。
- [ ] 分类/打标/排序辅助采用可回退策略（规则 baseline 可独立运行）。
- [ ] 模型供应商可切换（OpenAI/Anthropic/MiniMax）且配置隔离。
