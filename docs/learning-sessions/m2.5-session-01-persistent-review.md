# M2.5 学习复盘 01：持久化审核指令与 pending 复检发布

## 1. 本次实现了什么
- 新增审核指令存储抽象与文件实现：`src/review/instruction-store.ts`。
- 周报审核节点改为“持久化指令优先，CLI 参数 fallback”。
- CLI 新增 `--review-instruction-root`、`--recheck-pending`、`--report-date` 参数。
- 新增 pending 周报复检流程：读取 review 快照，复用审核/发布节点，按状态回写 review/published 产物。
- review JSON 新增 `snapshot` 字段，确保复检时不重跑采集链路也能重建报告。

## 2. 流程图（M2.5 增强）
```text
日常 run:
START
  -> collect_items
  -> normalize_items
  -> dedupe_items
  -> classify_items
  -> rank_items
  -> build_outline
  -> review_outline   (持久化指令优先，CLI fallback)
  -> review_final     (持久化指令优先，CLI fallback)
  -> publish_or_wait
  -> build_report
  -> END

pending 复检:
load_snapshot
  -> review_outline
  -> review_final
  -> publish_or_wait
  -> build_report
  -> persist_outputs
```

## 5. 3 分钟复盘模板（M2.5 版本）
```text
【1】本轮目标（30s）
- 我本轮要验证的能力点是：审核动作与执行任务解耦后，系统是否还能稳定完成“人工审核优先 + 截止兜底发布”。
- 我完成后的可见结果是：周报支持持久化审核指令、支持 pending 复检发布、状态与文案保持一致。

【2】关键实现（60s）
- 我改了哪 3 个核心文件：
  1) `src/review/instruction-store.ts`
  2) `src/pipeline/nodes.ts`
  3) `src/cli.ts`（含 `src/pipeline/recheck.ts` 复检执行）
- 每个文件“为什么要改”：
  - `instruction-store`：把审核输入抽象成可替换存储，先用文件实现最小闭环。
  - `nodes`：把审核判断统一为“持久化优先，CLI fallback”，保证流程可用且向后兼容。
  - `cli/recheck`：补复检入口，从 snapshot 重建状态，避免重采集导致发布内容漂移。

【3】运行验证（45s）
- 我执行的命令：
  - `pnpm test`
  - `pnpm build`
  - `pnpm run:weekly:mock`
  - `npx tsx src/cli.ts run --mode weekly --recheck-pending --report-date 2026-03-05 --generated-at 2026-03-09T05:00:00.000Z`
- 结果是否符合预期：符合；测试 15/15 通过，复检可触发 `timeout_published` 并落盘到 review/published。
- 有无 warning/边界场景：有，旧版无 snapshot 的历史周报无法复检；当前会显式报错提示先用新版 run 生成。

【4】设计取舍（30s）
- 我放弃了哪个方案，原因是：放弃“复检时重跑采集链路”，因为会引入内容漂移，破坏审核版本与发布版本一致性。
- 当前实现的风险点是：文件存储并发写冲突与人工编辑 JSON 质量问题；下一阶段应迁移 DB/API 并加审计与锁。

【5】下一步（15s）
- 我下一轮最小可执行目标是：增加定时守护任务，定时扫描 pending 周报并在截止点自动触发 `--recheck-pending`。
```
