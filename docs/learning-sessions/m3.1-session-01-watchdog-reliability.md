# M3.1 学习复盘 01：watchdog 锁 + 重试 + 告警输出

## 1. 本次实现了什么
- watchdog 增加单机 lock 文件互斥，避免重复巡检。
- watchdog 增加单候选复检重试（次数与间隔可配置）。
- watchdog 增加 summary 落盘（`outputs/watchdog/weekly/*.json`）与失败 alert 日志。
- 新增锁机制与重试相关单测，覆盖冲突、重试成功、重试失败场景。

## 2. 流程图（M3.1）
```text
watch_pending_weekly
  -> acquire lock
  -> scan review artifacts
  -> filter pending_review + pending
  -> recheck with retry
  -> optional persist (skip when dry-run)
  -> write summary file
  -> alert when failed > 0
  -> release lock
```

## 3. 源码导读（建议阅读顺序）
1. `src/utils/file-lock.ts`
   - 看 `acquireFileLock`，理解 `wx` 原子锁为何可避免并发重复执行。
2. `src/pipeline/watchdog.ts`
   - 看 `runRecheckWithRetry`，理解重试循环与失败计数如何设计。
   - 看 `runPendingWeeklyWatchdog`，理解 skipped/processed/published/failed 统计口径。
3. `src/cli.ts`
   - 看 `runWatchPendingWeekly`，理解锁、扫描、执行、summary 落盘的串联方式。
   - 看 `persistWatchdogSummary`，理解监控数据如何沉淀到文件。

## 4. 验证结果
- `pnpm test`：通过（23 tests）。
- `pnpm build`：通过。
- `pnpm run:weekly:mock`：通过。
- `npx tsx src/cli.ts run --mode weekly --watch-pending-weekly --dry-run --generated-at 2026-03-09T05:00:00.000Z`：通过，生成 summary 文件。
- 锁冲突验证：预写 lock 文件后执行 watchdog，输出 `[watch-skip] lock already held`。

## 5. 3 分钟复盘模板（M3.1 版本）
```text
【1】本轮目标（30s）
- 我本轮要验证的能力点是：定时守护在真实运行中能否做到“不会并发重复执行、遇到瞬时错误可恢复、结果可追踪”。
- 我完成后的可见结果是：watchdog 增加 lock + retry + summary，执行结束可直接审计每次巡检结果。

【2】关键实现（60s）
- 我改了哪 3 个核心文件：
  1) `src/utils/file-lock.ts`
  2) `src/pipeline/watchdog.ts`
  3) `src/cli.ts`
- 每个文件“为什么要改”：
  - `file-lock.ts`：把互斥能力做成独立模块，便于复用和单测。
  - `watchdog.ts`：将重试逻辑放在执行层，避免 CLI 里出现复杂控制流。
  - `cli.ts`：统一接入参数、lock 生命周期、summary 落盘和 alert 输出。

【3】运行验证（45s）
- 我执行的命令：
  - `pnpm test`

【4】设计取舍（30s）
- 我放弃了哪个方案，原因是：放弃引入分布式锁中间件，当前单机部署下 lock 文件更简单且足够。

【5】下一步（15s）
- 我下一轮最小可执行目标是：把 watchdog 失败项接入外部告警通道（如 webhook）并支持非 0 退出码策略。
```
